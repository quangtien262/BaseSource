Introduction

This document aims to explain the best way to work with the product and its components.

- All the best and enjoy coding.

Getting started tips

  • Do not start from scratch, use an existing asset and modify it to learn how it works.
  • Explore the sources for ideas and sample code.
  • Use Firebug or Chrome Developer Tools to find bugs on your website. Using one of those tools will help you to save time analyzing the site and finding elements structure, like classes, id or tags
    • Quick tip: open your site with Chrome, press F12 and go to console tab, reload your page and if something goes wrong you will see your page errors in red text.
  • In case of errors, someone might have seen it too, you can try a Google search for a quick fix.

Starting the app

Since this is AngularJS based application you need a server (Apache, IIS, xampp, etc) to serve the html files and perform http request to load all views.

Important Opening the index.html with a double click (i.e. using file:// protocol) will show you only a blank page because there’s no server that response to the requests made for each view in order to display the app interface.

Basic server setup (requires nodejs)

This is simple solution for a basic server setup using nodejs that might help you. If you don't have installed nodejs take a look at the Build section to learn about it

  • Install the http-server (-g installs globally)

    npm install http-server -g

  • Move to the root folder of the theme (where index.html is located) and run

    http-server . -a 127.0.0.1 -p 8080

  • If everything goes fine you can now access to the app at http://127.0.0.1:8080/

Main folders explanation

app/

This folder contains the assets like images, fonts, etc and compiled files. This files are ready to deploy on your server.

  • views/ This folder contains the compiled html files for the views and partials used for the app.
  • css/ This folder contains the compiled stylesheets.
  • js/ This folder contains only the final build of javascript files placed in master/ folder. Such final build is generated using Gulp. By default all Javascript are linked to files in /master/components folder.

master/

This folder contains the source files organized by components. Each component can contains JADE/HTML, LESS and JS files divided into modules.
This folder also will contain the node_modules folder after run "> npm install"

vendor/

This folder contains the vendor files used to include plugins and other components. This folder is handled using Bower, optionally, you can remove or upgrade the vendor file using such tool.
There's no bower_components, all files installed with bower are placed directly into this folder.

server/

This folder contains server side files used for demonstration. In a real app your server should listen to specific address and return the content of this files (and/or others) with specific information.

Build

Important You only need to follow this instructions if you plan to use build tools.

Build Tools

Node.js is a platform built on Chrome’s JavaScript runtime for easily building fast, scalable network applications.

Gulp is a task manager, you can define different tasks to run certain commands. Those commands does the compilation job for precompiled languages (JADE/LESS), and concatenates the JS files.

Bower is a dependency manager, it works by fetching and installing packages from all over, taking care of hunting, finding, downloading, and saving the stuff you’re looking for. Bower keeps track of these packages in a manifest file, bower.json.

The package includes under the master/ folder the file gulpfile.js, package.json and bower.json to install the required components in order to compile the source files.
.bowerrc includes directives for bower and is used to change the destination folder where files are downloaded.
.jshintrc contains directives used to ensure a well written Javascript code.

Installing tools

The following steps are intended to be an orientation guide, if you are not experienced with this you will need to learn a bit more about it from Google :)

  • To install node and npm, go to http://nodejs.org/
  • Download and install GIT for your platform from http://git-scm.com/downloads

Once you have all tools installed

  • Open a console/terminal and npm install -g gulp bower to install Bower and Gulp

  • Move to the package master/ folder, there run the following commands

    • This command will install all node dependencies. npm install.
    • This command will install vendor dependencies bower install

  • Finally run gulp to start the task manager.

If everything goes fine, you should see messages in your terminal telling you that all the task are done ok. Gulp will watch for file changes to compile them automatically.

Gulp configuration

If you need to change the configuration used by Gulp, mainly paths, you can do that by editing the file gulp.config.js

In such file you can also change Gulp plugin options.

Main Gulp tasks

This command will run the default task without minify assets

> gulp

This command will run the default task and start the built-in server using BrowserSync

> gulp serve

Compiles all assets for production and start a server running the app in the browser

> gulp serve-prod

This modifier will create source-maps when compiling assets

> gulp --usesourcemaps

This task will compile the application for distribution

> gulp dist

Delete build files

> gulp clean

Be careful the clean task will delete files without any confirmation or undo option.
Configured to be deleted: /app/css, /app/js, /app/views/, /app/img, /index.html

Lint your JavaScript code

> gulp lint

AngularJS Template Cache

To enable this feature, run gulp with the following command

> gulp --usecache

Note: This flag is recommended to be combined with 'build' task so you can generate the best optimized version for production environment.

Usage

This template was developed to use Gulp as a start of the development workflow.
Gulp provides plugins to serve the app, perform automatic browser reloading after any change and build of assets for distribution. It also allows to work with agile languages like JADE/LESS/etc and modular JavaScript (concatenated).

Beyond that, if you are not convinced about build tools, the current setup has a basic support to work with static files (html, css) without needing build tools.

We encourage you to use build tools, the benefits greatly outweigh the costs of learning.

Static files organization (no gulp)

The most simple usage is by pointing the server root the root folder of the package, where the index.html is placed and then start editing files within app/ folder.

This allows to work directly with raw files like CSS, HTML and JS without worrying about build tools.
Now, to edit files focus on the following locations:

  • Styles are accessible in folder /app/css
  • Markup is accessible in folder /app/views (raw html)
  • Scripts are accessible in folder /master/components

Optionally, you can move 'components' folder from 'master' to 'app' folder, then open/edit index.html and change the paths for all script from '/master/components' to '/app/components'

Compiled files organization

As quick reference, the current setup does the following compilations:

Language Produces
HTML HTML
JADE HTML
LESS CSS
Javascript Javascript (concatenation)

HTML files found in components folder are copied directly to the 'app' folder

Styles

Each component contains its own styles. Gulp read all folder recursively and includes all files related to styles into one single file: app.css

The inclusion order is determined by the way file are read like follows:

path_to_components + 'app.less',
path_to_components + '**/*.less'

As you can see, the file app.less is read first. Use this file to add any styles that you need before anything else. For example, @import directives.

Scripts

Each component contains its own scripts (module, services, controllers, etc). Gulp read all folder recursively and includes all files related to scripts into the index file using a loop (see index.jade)

The inclusion order is determined by the way file are read like follows:

path_to_components + 'app.module.js'
path_to_components + '**/*.module.js'
path_to_components + '**/*.js'

The main file app.module.js is the start point of the application and includes all modules related to components. Third party modules are included in the core component core/core.module.js

Views

By default views are written in JADE for quick development. Using this language you will get a well organized and readable markup without need to deal with closing tags and tag delimiters.

Gulp workflow also supports raw HTML, all this files are copied directly without any special process. Only JADE views are compiled separately into their respective HTML file.
Meaning,
file.jade compiles to file.html
file.html is copied as file.html

We use two great tools to build the index file: Usemin and Inject

  • The first one, Usemin, is used to combine a set of files into one single file. For example when you see // build:vendor(..) app/js/vendor.js it means all files withing this build block will be concatenated into the file vendor.js

Usemin is only executed when you run > gulp build

  • The second one, Inject, is used only to inject automatically the file templates.js, this file is only created when you run > gulp --usecache because it creates the angular templateCache file with all views markup in there.

Sidebar

Sidebar related styles, markup and scripts are contained under the components/sidebar folder.

The file sidebar.tpl.jade contains the markup to generate the sidebar and it's basically a loop that read the menus entry added from different modules. (see below about adding a new menu entry).

The sidebar uses an element (.sidebar-layout-obfuscator) to show a backdrop screen when it's open on mobiles. This backdrop also allows the user to click on almost any place to close the sidebar easily.

Header

Header (or top navbar) related styles, markup and scripts are contained under the components/header folder.

The file header.tpl.jade contains the markup to generate the top navigation bar.

Under this module also exists the search bar controllers. The markup for the search bar can be found in file header-search.tpl.jade

In the file header.controller.js you can find functions to show the search bar and the right sidebar. Both components are based on Bootstrap modals with minor modifications to the styles. Custom modals styles can be found in file common/modals.less

All related stuff for this layout and page content can be found under folder components/user

Other layouts

This folder contains example markup to generate different layouts for the main content.

RTL

RTL mode is supported using a Nodejs tool called rtlcss which inverts most the CSS properties to change the page orientation.

There's also a property $rootScope.app.layout.rtl to know when the site is in RTL mode.

The enable RTL you need the RTL version of the template and Bootstrap stylesheets.

    <link rel="stylesheet" href="path/to/bootstrap-rtl.css">
    <link id="appcss" rel="stylesheet" href="app/css/app-rtl.css">

RTL version of the main application stylesheet is generated by Gulp and it's named app-rtl.css

Bootstrap doesn't have an official RTL version but you can generate it using the mentioned tool, rtlcss.
To do that, you need nodejs installed on your machine. After that install the tool with npm like is described here

Once you have the tool ready to use, download or go the vendor/bootstrap folder and look for the file bootstrap.css.
Open a console/terminal in the same folder of that file and run the following command

$ rtlcss bootstrap.css bootstrap-rtl.css

When the command ends you should have a new file bootstrap-rtl.css in the same folder ready to include in your index file.

Router

The router module, under folder components/router contains all related stuff for custom functions used to add new routes.

The RouterProvider is used to provide LazyLoading to the application by wrapping the new routes addition and to help with common task like requesting vendor assets per route (lazy load)

This example shows as example the Dashboard route with comments on main parts:

Note that 'Router' is the mentioned provider and is injected in order to use it.

(function() {
    'use strict';

    angular
        .module('app.dashboard')
        .run(dashboardRoute);

    // Inject the RouterProvider
    dashboardRoute.$inject = ['Router'];

    function dashboardRoute(Router) {

        // Usage is the same like the UIRouter
        // First param is the route state name and second param are state options
        Router.state('app.dashboard', {
            url: '/dashboard',
            title: 'Dashboard',
            // No need to add here the base path for each view every time
            templateUrl: 'dashboard.html',
            // The special 'require' property allows to load vendor assets before loading the route
            require: ['angular-flot', 'easypiechart', 'sparkline', 'vector-map', 'vector-map-maps']
            // resolve: can also be used
        });
    }

})();

The templateUrl has a special treatment, any file added here will be appended to the base path configured for each view.

The require property is converted to a normal resolve or mixed if present. Each value in the array of require corresponds to an entry in the constant APP_REQUIRES defined in file router.constants.js, this constant is used to list the paths for each vendor assets.
This technique is useful to have all vendor paths in one place, and it also saves time when the same paths are used by more than one route.

The RouterProvider can be configured in file router.provider.js

var config = {
    // The paths where html template resides
    viewsBasePath: 'app/views/',
    // Automatically prepend views path to all templatesUrl?
    useViewsBasePath: true,
    // Set the following to true to enable the HTML5 Mode
    // You may have to set <base> tag in index and a routing configuration in your server
    html5Mode: false,
    // defaults to dashboard
    defaultRoute: '/app/dashboard'
};

Translation

The translation system uses the AngularUI Translate module. This modules simplifies the translation system by loading translate references from a JSON file and replacing the content where the reference has been used.

<h3 ui-translate="reference.NAME">Text that will be replaced</h3>

<h3>{{ 'reference.NAME' | translate }}</h3>

<a href="#" title="{{ 'reference.NAME' | translate }}">Link</a>

Example content of the file server/i18n/en.json

{
  "WELCOME": "Welcome",
  "header": {
    "SEARCH": "Search.."
  }
}

As a convention, we use the key name with the translated text in uppercase.

Customization

Here you will find information about how to add new features to the template.

Adding new Components

To add a new component create a new folder under 'master/components/'

For example,'master/components/mycomp'

Adding styles

To add a new styles just a create a new LESS file into the component folder. This file will be automatically compiled and added to final CSS files.

Separated components should not have conflicts between CSS rules but just in case, check the specificity in your styles (read more here)

Adding views

Create a new JADE (or HTML) file with the name your route expect to find the file.

For example, mycomp.jade will be compiled to mycomp.html

The minimum necessary for a view markup is the .container-* element

mycomp.jade

.container-fluid
    // here goes the content, .row, .panels., .cards, etc...

or, with a centered container 

.container.container-lg
    // here goes the content, .row, .panels., .cards, etc...

a raw html version of above looks like

<div class="container container-lg">
    <!-- here goes the content, .row, .panels., .cards, etc... -->
</div>
Adding a Module

Your new component should defined its own module in order to keep the app modular. Place in the new folder a module file with the name mycomp.module.js and the content could be:

(function() {
    'use strict';

    angular
        .module('myapp.mycomp', []);
})();

All modules should be added as a dependency of the main app, in file app.module.js Example

(function() {
    'use strict';

    angular
        .module('centrict', [
            'app.core',
            ...
            'myapp.mycomp'
        ]);
})();
Adding a Route

Add the file mycomp.route.js with the following example content

(function() {
    'use strict';

    angular
        .module('myapp.mycomp')
        .run(mycompRoute);

    mycompRoute.$inject = ['Router'];

    function mycompRoute(Router) {

        Router.state('app.myroute', {
            url: '/myroute',
            title: 'My Route Title',
            templateUrl: 'mycomp.html'
            // optionally add require and resolve here
        });
    }

})();

Note that the route name is 'app.myroute' in order to be handled by the main router name 'app'. This route state is defined in core.route.js.

Adding a Menu entry

To add a new menu, you can use the Menu Service(under components/menu).
This menu service collects all menu items added from each component and is used by the sidebar to build recursively each entry.

Create a file named mycomp.menu.js with the following example content:

(function() {
    'use strict';

    angular
        .module('myapp.mycomp')
        .run(mycompRun);

    mycompRun.$inject = ['Menu'];

    function mycompRun(Menu) {

        var menuItem = {
            name: 'My Comp',    // the menu name displayed
            sref: 'app.mycomp', // the target state
            order: 1,           // the order in the list of menu items
            iconclass: 'ion-chevron-right' // the classname for the icon
        };

        Menu.addItem(menuItem);

    }
})();

Check other components for examples to build a menu with subitems.

Adding vendor assets

To add a new vendor assets you need to follow 3 steps:

1- Download and install.

Usually this is done with 'bower install 3rd-party-plugin'.

All files downloaded will be placed in vendor/ folder

2- Add main files to the application.

Here are 2 options:

  • Files available across the entire application
    • Add the main files directly to the index file
  • Files available to certain routes (lazyloaded)
    • Add the paths to the APP_REQUIRE constant, in components/router/router.constants.js
    • Use the constant entry name in your route 'require' property
